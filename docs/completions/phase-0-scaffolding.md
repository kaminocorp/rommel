## Why each file

| File | Why |
|---|---|
| `.gitignore` | Covers all four toolchains in one place: Node (`node_modules/`, `.next/`), Python (`__pycache__/`, `.venv/`), Go (`sandbox-daemon/dist/`), and deploy tooling (`.fly/`, `.vercel/`). Generated proto clients (`proto/clients/*/{src,gen}/`) are gitignored — codegen is reproducible from `proto/schemas/`, so the artifacts shouldn't be committed. |
| `.editorconfig` | Cross-editor consistency without forcing everyone onto the same editor. 2-space default, 4-space for Python/Go, tab for Makefile (required by `make`), no trailing-whitespace trim on Markdown (preserves intentional double-space line breaks). |
| `README.md` | One-paragraph orientation per the plan, with a table of subtrees and pointers into `docs/`. Deliberately doesn't duplicate `vision.md`. |
| `Makefile` | Top-level **router**, not a build system. Each target (`lint`, `test`, `build`, `bootstrap`, `clean`) delegates into the per-subtree Makefile via a `run_if_exists` helper. The helper no-ops with a friendly note when the subtree doesn't yet exist — this keeps CI green during the multi-phase scaffolding rollout. |
| `package.json` | Minimal pnpm workspace root. `"private": true` (never published), pinned `packageManager` (Corepack reads this), and engines specified so pnpm/Node version drift is caught early. No runtime deps — that's intentional, this file exists to anchor the workspace. |
| `pnpm-workspace.yaml` | Lists the two TS workspaces that will exist: `frontend/` and `proto/clients/ts/`. pnpm tolerates missing globs, so this is safe to commit before the dirs exist. |
| `.nvmrc` | Pins Node 20 for editors/CI that respect it. |
| `.tool-versions` | asdf/mise pin for Node 20.18.0, Go 1.22.8, Python 3.12.7, pnpm 9.12.0. Single source of truth so all four toolchains stay aligned across machines. |
| `.github/workflows/*.yml` | Path-filtered CI per the plan. Each workflow gates on a sentinel file (`frontend/package.json`, `backend/pyproject.toml`, `sandbox-daemon/go.mod`, `proto/codegen.sh`) and skips cleanly if absent — so workflows are wired correctly today and "wake up" the moment their subtree lands. |

## Decisions made (and the alternatives ruled out)

### Bare pnpm workspaces, not Turborepo
`techstack.md` left this open. Picked bare for v1 — Turborepo's value (remote caching, task graphs) doesn't pay off until there are multiple TS packages doing real work. Easy to layer on later if cross-package builds get slow.

### CI workflows are defensive (gate-and-skip), not "wait to add"
The alternative was leaving `.github/workflows/` empty until each subtree exists. Rejected because: (a) once subtrees start landing, the friction of "did I remember to add the workflow?" causes drift; (b) wiring path filters once now means the very first PR that touches `frontend/` triggers exactly the right job. The gate-and-skip pattern keeps the matrix structurally complete from day one.

### Makefile uses `run_if_exists` instead of hard-coded targets per subtree
Same reasoning. Adding a subtree is a single `mkdir` + per-subtree Makefile away from being picked up by the root router. No edits to the root `Makefile` needed when a subtree appears.

### `proto/clients/*` generated dirs are gitignored
Per the plan, `make proto` regenerates them; the `proto.yml` workflow fails CI if the regenerated output diverges from committed schemas. This avoids the classic "generated code committed for convenience, then drifts from schemas" footgun.
